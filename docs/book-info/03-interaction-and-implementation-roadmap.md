# Interaction And Implementation Roadmap

## 目标

先把“书籍信息”做成真正可用的功能，再慢慢把它做得越来越有意思。

所以路线要分层：

- 先做日常高频
- 再做编辑能力
- 最后做封面集、洞察与分享

## 交互主线

书籍信息最重要的入口不是书库卡片本身，而是下面这些时刻：

1. 从书库进入一本书
2. 从阅读器顶部书名进入
3. 从笔记/高亮反查到书
4. 从统计页点击一本主读书
5. 从搜索结果进入

这意味着它不能只是一页独立页面，而应该有三种形态。

### 形态 A：完整页面

适用于：

- 书库入口
- 搜索入口
- 移动端主入口

### 形态 B：阅读器侧边抽屉

适用于：

- 桌面端阅读器中查看书籍信息
- 不打断阅读的快速查看与编辑

### 形态 C：快速预览

适用于：

- 书库 hover
- 长按卡片
- 右键菜单

## 关键交互细节

### 1. 继续阅读必须是一号动作

无论移动端还是桌面端，最显眼的都必须是 `继续阅读`。

### 2. 评分不能藏太深

评分与书评应该紧挨标题区，不要埋到更多菜单。

### 3. 封面编辑要“轻”

用户切换封面不应该像编辑元数据那样沉重。

建议：

- 点封面即进入封面集
- 点某个封面即预览
- 再点“设为主封面”

### 4. 书评要低门槛

不要逼用户一上来写长评。

建议先给：

- 一句话短评
- 星级评分

长评作为下一层展开。

### 5. 版本页要服务高级用户

把以下能力都放进去：

- 重新抓取元数据
- 查看文件来源
- 替换封面
- 修正元数据
- 查看同步状态

## 与现有系统的联动

### 与阅读器

- 从阅读器标题进入书籍信息
- 从书籍信息一键回到当前位置
- 可从版本页修改封面与元数据

### 与统计

- 本书阅读时长、字数、速度
- 本书目标
- 阅读时间线

### 与笔记

- 直接聚合本书高亮、笔记、书签
- 支持导出与 AI 整理

### 与 TTS

- 如果本书支持朗读，显示：
  - 听书入口
  - 上次听到的位置
  - 定时器状态

### 与 AI

- 问这本书
- 总结本书
- 提炼人物 / 概念 / 主题
- 帮我写一句短评

## 分阶段实施

## Phase 0：冻结方案

产出：

- 本目录三份文档

目标：

- 明确信息架构
- 明确编辑边界
- 明确首版范围

## Phase 1：书籍信息 MVP

目标：

- 建立可访问的“书籍主页”

范围：

- Hero
- 继续阅读 / 听书 / 问 AI
- 概览 tab
- 阅读 tab
- 笔记 tab

- 编辑评分

## Phase 2：可编辑元数据与封面

目标：

- 让用户真正能整理自己的书

范围：

- 元数据修订
- 主封面替换
- 封面集
- 封面来源标记

## Phase 3：书评与洞察

目标：

- 让这页开始有记忆感和沉淀感

范围：

- 长评
- 书籍 DNA
- AI 摘要
- 人物 / 概念 / 主题卡

## Phase 4：预览与抽屉

目标：

- 让书籍信息嵌入整个产品

范围：

- 书库 hover 预览
- 阅读器右侧抽屉
- 搜索结果快捷信息

## Phase 5：封面集与分享

目标：

- 把“拥有一本书”的乐趣做出来

范围：

- 封面集导入/导出
- 本书海报
- 封面集分享卡
- 评分与短评分享卡

## 当前建议的正式开工顺序

最稳的顺序：

1. 新建 book info view model
2. 定义 user editable fields
3. 实现页面壳子
4. 接入评分与阅读状态
5. 接入笔记聚合
6. 接入版本页
7. 接入封面替换
8. 做封面集
9. 做 AI 洞察

## 验收标准

### MVP 验收

- 用户能从书库、阅读器进入书籍信息
- 用户能继续阅读、听书、问 AI
- 用户能看到进度、时长、字数、笔记摘要

### 编辑能力验收

- 用户能改评分
- 用户能写一句话短评
- 用户能修改主封面
- 用户能修正关键元数据

### 封面集验收

- 一本书可拥有多张封面
- 用户可切换主封面
- 用户可删除、排序、命名封面

## 下一步建议

我建议这套文档落定后，直接继续做两件事：

1. `docs/book-info/prototype/`
   - 出移动端和桌面端 HTML 交互稿
2. `book-info schema draft`
   - 把 `book_meta_override / book_cover_asset / book_review` 的正式 TypeScript 类型先写出来

这样后面不管是先做 UI，还是先做数据层，都不会返工太大。
